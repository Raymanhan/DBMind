# 发明专利技术交底书

---

## 一、发明名称

**一种基于AI模型感知的数据库智能交互方法及系统**

---

## 二、技术领域

本发明涉及数据库管理与人工智能交叉技术领域，具体涉及一种利用大语言模型（LLM）对数据库模式（Schema）进行感知理解，实现自然语言到SQL语句的智能生成、SQL语句优化分析与数据库智能交互的方法及系统。

---

## 三、背景技术

随着数据规模的增长，数据库操作日益复杂。传统数据库管理工具存在以下技术问题：

1. **SQL编写门槛高**：用户需要掌握SQL语法和数据库模式（包括表名、字段名、字段类型、表间关系等），对非专业用户形成较高的技术壁垒。

2. **模式理解依赖人工记忆**：在拥有数十乃至数百张表的大型数据库中，开发人员和数据分析师难以记忆所有表结构和字段含义，导致SQL编写效率低下、容易出错。

3. **SQL优化缺乏智能辅助**：传统工具仅提供语法高亮和自动补全，无法对SQL查询进行语义级的性能分析和优化建议。

4. **现有AI方案上下文割裂**：已有的NL2SQL（自然语言转SQL）方案通常将数据库模式信息以静态方式注入提示词，无法动态感知用户当前工作上下文（如编辑器中的SQL、用户提及的表、查询历史等），导致生成结果不够准确。

5. **交互模式单一**：现有方案多为"输入问题→输出SQL"的单轮交互，缺乏多轮对话、上下文记忆、表引用动态解析等智能交互能力。

因此，亟需一种能够动态感知数据库模式、结合用户工作上下文的AI驱动数据库智能交互方法和系统。

---

## 四、发明目的

本发明的目的在于克服上述现有技术的不足，提供一种基于AI模型感知的数据库智能交互方法及系统，实现以下技术目标：

1. 动态构建数据库模式索引，实时感知表结构、字段信息、注释和表使用频率；
2. 基于用户在交互过程中提及的表引用（@database.table），自动解析并注入相关模式上下文至AI模型；
3. 实现自然语言到SQL的智能生成，结合当前编辑器SQL和对话历史进行多轮上下文感知交互；
4. 提供SQL智能优化分析和执行计划可视化解读；
5. 构建完整的"模式感知→上下文构建→AI推理→结果交互"的闭环智能交互流程。

---

## 五、技术方案

### 5.1 系统总体架构

本系统采用前后端分离的桌面应用架构，包括以下核心层：

**前端交互层**：基于React框架构建用户界面，包含SQL编辑器（Monaco Editor）、AI对话面板、数据库模式树、查询结果网格等模块。

**后端服务层**：基于Rust语言和Tauri框架构建，通过进程间通信（IPC）与前端的交互层进行数据交换，包含以下核心子系统：

- **模式索引子系统（dbmind-schema）**：负责数据库模式的采集、存储、索引和检索；
- **AI上下文引擎子系统（dbmind-ai）**：负责AI上下文的构建、提示词组装和多模型适配；
- **SQL处理子系统（dbmind-sql）**：负责SQL语句的分割、校验、格式化、表引用提取和标识符处理；
- **查询执行子系统（dbmind-query）**：负责SQL语句的执行、结果缓存和查询取消；
- **结果缓存子系统（dbmind-cache）**：提供分级存储策略的查询结果缓存。

### 5.2 核心方法流程

本发明的核心方法包含以下步骤：

#### 步骤S1：数据库模式索引构建

系统连接数据库后，自动采集数据库的完整模式信息，具体包括：

**(1) 模式数据采集**：针对MySQL数据库，执行`information_schema`查询获取表名、表类型、表注释、行数、列名、列类型、是否可空、是否主键、默认值、列注释、最大长度、小数位数等元数据；针对PostgreSQL数据库，额外采集索引信息和外键关系。

**(2) 模式索引存储**：将采集的模式数据以`{数据库名}.{表名}`为键，存入内存中的读写锁保护的模式索引（SchemaIndex）中，支持并发安全访问。

**(3) 使用频率统计**：记录每张表的访问频次（usage_scores），用于后续按使用频率排序，优先推荐高频使用的表。

**(4) 跨库搜索**：支持跨数据库的全局表搜索，按表名、表注释、数据库名进行模糊匹配。

#### 步骤S2：AI上下文动态构建

当用户发起AI交互请求时，系统执行以下上下文构建流程：

**(1) 表引用检测与解析**：
- 用户在AI对话输入框中通过`@数据库名.表名`格式引用数据库表；
- 系统实时检测`@`符号触发，展示跨数据库的表搜索下拉列表；
- 用户选择或输入表名后，系统调用模式索引获取该表的完整DDL定义；
- 将解析到的表DDL添加至当前对话的"已固定表"（pinnedTables）列表。

**(2) 粘贴感知处理**：
- 当用户粘贴包含`@数据库名.表名`引用的文本时，系统自动检测并立即解析引用，将对应表DDL加入上下文，无需用户手动触发。

**(3) 当前编辑器SQL感知**：
- 系统自动读取当前SQL编辑器中正在编辑的SQL语句；
- 对该SQL进行AST解析，提取其中引用的表名；
- 对提取到的表名，自动从模式索引中获取其DDL定义并加入上下文。

**(4) 上下文优先级策略**：
- 当用户明确引用了表（通过@提及），仅使用用户引用的表构建上下文；
- 当用户未引用任何表，系统根据使用频率统计（get_most_used）选取当前数据库中使用频率最高的前N张表作为默认上下文；
- 合并已固定表DDL和自动提取表DDL，去重后形成完整的模式上下文。

**(5) AI上下文包（AiContextBundle）生成**：
系统将以下信息组装为结构化上下文包：
- **模式摘要（SchemaSummary）**：包含相关表的列名、列类型、列注释、行数等信息；
- **当前编辑器SQL**：用户正在编辑的SQL语句；
- **错误信息**：如有执行错误，包含错误描述；
- **数据库名称**：当前操作的数据库标识。

#### 步骤S3：AI提示词智能组装

系统根据不同的交互场景，采用差异化的提示词组装策略：

**(1) 自然语言转SQL（NL2SQL）场景**：
- 在提示词中注入完整的相关表模式信息，格式为`{数据库名}.{表名}`下逐列列出列名、数据类型、注释和行数；
- 明确约束AI模型：仅使用模式中存在的列，所有表引用必须使用完全限定名（`数据库.表`），使用对应数据库类型的标识符引用风格；
- 将用户自然语言问题作为待转换输入。

**(2) SQL智能对话场景**：
- 注入系统角色提示词，定义AI为数据库SQL专家助手；
- 根据数据库驱动类型（MySQL/PostgreSQL）注入对应的SQL方言说明；
- 注入已固定表的DDL定义作为模式参考上下文；
- 注入当前编辑器SQL作为待分析对象；
- 注入完整的多轮对话历史，实现上下文连续的智能对话。

**(3) SQL优化分析场景**：
- 构建分析提示词，引导AI模型从查询目的、涉及表、关键过滤条件和连接条件、性能注意事项等维度进行SQL分析。

**(4) 方言自适应**：
- 系统自动检测数据库类型（MySQL或PostgreSQL），在系统提示中注入对应的SQL方言规范：
  - MySQL：反引号标识符、AUTO_INCREMENT、ON DUPLICATE KEY UPDATE、IFNULL()等；
  - PostgreSQL：双引号标识符、SERIAL/BIGSERIAL、RETURNING *、ON CONFLICT、ILIKE等。

#### 步骤S4：AI模型流式推理与结果处理

**(1) 多模型适配**：
- 系统定义统一的AI提供者接口（AiProvider），支持OpenAI API兼容的所有模型服务；
- 支持配置自定义API端点（api_url），兼容Ollama等本地模型服务；
- 支持配置模型名称、最大Token数、温度参数等模型参数。

**(2) 流式推理**：
- 采用Server-Sent Events（SSE）协议接收AI模型的流式响应；
- 解析SSE数据流中的增量内容（delta content），实时通过事件系统推送至前端；
- 前端实时渲染AI响应内容，实现打字机效果的流式输出体验。

**(3) SQL代码块提取与交互**：
- 前端解析AI响应中的` ```sql ``` `代码块，渲染为独立的可交互SQL组件；
- SQL代码块提供语法高亮显示（关键字、字符串、数字分色）；
- 提供"一键执行"按钮：点击后将SQL注入编辑器新标签页并自动执行；
- 提供"一键复制"按钮：将SQL复制至系统剪贴板。

#### 步骤S5：SQL安全处理与执行

**(1) SQL安全校验**：
- 在执行前对SQL进行安全规则检查，识别危险操作模式：
  - 检测DROP TABLE、DROP DATABASE、TRUNCATE等破坏性DDL操作；
  - 检测ALTER TABLE ... DROP等结构修改操作；
  - 检测无WHERE条件的DELETE和UPDATE操作；
- 发现危险操作时生成警告信息。

**(2) SQL语句分割**：
- 使用SQL解析器（sqlparser）对多语句SQL进行AST级分割；
- 正确处理字符串字面量、行注释（--）、块注释（/* */）中的分号，避免误分割；
- 支持基于光标位置的当前语句检测。

**(3) 标识符自动修正**：
- 对包含连字符（-）的未引用标识符自动添加反引号引用；
- 正确区分SQL字符串字面量、注释和标识符中的特殊字符，避免误处理。

**(4) 分级结果缓存**：
- 对查询结果采用分级存储策略：
  - 小于10,000行的结果存储于内存HashMap中；
  - 大于等于10,000行的结果存储于本地SQLite数据库中，按1,000行分块存储；
- 支持按单元格范围（行区间×列区间）的按需分块读取，实现大规模结果集的虚拟化滚动。

### 5.3 关键技术特征

**特征一：基于表引用的模式感知上下文注入机制**

系统创新性地设计了`@数据库.表`的表引用语法，用户在自然语言输入中通过该语法显式指定AI需要感知的表。系统检测到引用后：
1. 从模式索引中检索对应表的完整DDL；
2. 将DDL固定到当前对话会话中（pinnedTables），在整个对话生命周期内持续生效；
3. 支持粘贴感知：粘贴含引用的文本时自动触发解析。

该机制解决了现有方案"静态注入全库模式"导致的上下文过长、模型注意力分散的问题，实现了按需精准注入。

**特征二：基于使用频率的智能默认上下文选择**

当用户未显式引用任何表时，系统不盲目注入全库模式，而是基于历史表访问频率统计（usage_scores），选取使用频率最高的前N张表作为默认上下文。该策略在保证AI理解准确性的同时，有效控制了上下文长度，降低了模型推理成本。

**特征三：多源上下文融合的提示词构建方法**

系统将以下多源信息融合构建AI提示词：
- 系统角色定义（含数据库方言规范）；
- 用户固定表的DDL定义；
- 当前编辑器SQL（及其自动提取的表引用DDL）；
- 多轮对话历史。

这种多源融合策略确保AI模型在每次推理时都能获得充分且精准的上下文信息。

**特征四：SQL方言自适应的跨数据库支持**

系统根据连接的数据库类型自动切换SQL方言规范，在系统提示中明确指定目标数据库的语法规则（标识符引用风格、函数差异、DDL语法等），确保AI生成的SQL在不同数据库类型下均可正确执行。

**特征五：AI响应中SQL的可交互渲染**

系统创新性地将AI响应中的SQL代码块渲染为独立的可交互组件，支持一键执行和一键复制，打通了"AI生成SQL→用户确认→执行查看结果"的完整工作流。

### 5.4 系统模块间数据流

完整的智能交互数据流如下：

```
用户输入自然语言问题（含@db.table引用）
    │
    ▼
前端：提取@引用 → 调用IPC获取DDL → 固定到对话
前端：读取当前编辑器SQL → 提取表引用 → 获取DDL
前端：合并固定DDL + 自动提取DDL + 对话历史
    │
    ▼ IPC调用
后端：构建系统提示词（角色 + 方言 + DDL上下文 + 编辑器SQL）
后端：组装消息列表（系统消息 + 对话历史）
后端：调用AI提供者接口 → 流式SSE推理
    │
    ▼ 事件推送
前端：实时渲染AI响应 → 提取SQL代码块
前端：用户点击"执行" → 注入编辑器 → 执行SQL
    │
    ▼ IPC调用
后端：SQL安全校验 → 标识符修正 → 语句分割 → 执行
后端：结果分级缓存 → 返回元数据
    │
    ▼ 事件推送
前端：虚拟化数据网格展示查询结果
```

## 六、有益效果

与现有技术相比，本发明具有以下有益效果：

**（1）显著降低SQL编写门槛**：非专业用户可通过自然语言描述数据需求，系统自动生成符合数据库模式的准确SQL语句，无需记忆表结构和SQL语法。

**（2）提高SQL生成准确性**：通过@表引用机制和自动表引用提取，实现了精准的模式上下文注入，避免AI模型"幻觉"出不存在的列名或表名，大幅提升SQL生成准确率。

**（3）提升数据库操作效率**：一键执行AI生成的SQL、查询历史回放、多语句批量执行等功能，将传统需要多步骤手动操作的工作流压缩为单一交互动作。

**（4）智能上下文连续性**：多轮对话历史持久化和表引用固定机制，使AI能够在整个对话会话中保持对数据库模式的理解一致性，支持复杂查询的渐进式构建。

**（5）跨数据库兼容性**：SQL方言自适应机制使同一套AI交互流程能够无缝支持MySQL和PostgreSQL等不同数据库类型，无需用户关心语法差异。

**（6）安全可控的SQL执行**：SQL安全校验机制在执行前自动识别和警告危险操作，有效防止误操作导致的数据丢失。

**（7）高性能大规模数据处理**：分级结果缓存策略和虚拟化数据网格，支持流畅展示大规模查询结果，解决了传统工具在处理大数据集时的性能瓶颈。

## 八、具体实施方式

下面结合本发明的技术方案，描述一个典型的具体实施例。

### 实施例：用户通过自然语言查询数据库

**场景**：用户连接到一个MySQL数据库auth-cloud，需要查询某个部门所有员工的加班时长汇总。

**步骤1：模式索引构建**

系统连接数据库后，执行information_schema查询，获取auth-cloud数据库下所有表的模式信息，包括sys_user、sys_user_dept、daily_log_detail等表的列名、类型、注释、行数等，存入模式索引。

**步骤2：用户发起AI对话**

用户在AI对话输入框中输入：

> 查询"张三"所在部门所有员工2026年5月的加班总时长，按加班时长降序排列 @auth-cloud.sys_user @auth-cloud.sys_user_dept @auth-cloud.daily_log_detail

系统实时检测到@符号，展示表搜索下拉列表，用户选择三张表后，系统：
1. 从模式索引获取三张表的DDL定义；
2. 将DDL固定到当前对话会话。

**步骤3：AI上下文构建**

系统自动执行：
1. 读取当前SQL编辑器内容（如为空则跳过）；
2. 合并固定表DDL；
3. 从对话历史中提取之前的用户和AI消息；
4. 组装完整的AI上下文包。

**步骤4：AI提示词组装**

后端构建如下消息序列发送至AI模型：

- **系统消息**：角色定义（SQL专家）+ MySQL方言规范 + auth-cloud数据库标识 + 三张表的DDL定义（含列名、类型、注释）+ 当前编辑器SQL（如有）
- **用户消息**：历史对话 + 当前自然语言问题

**步骤5：AI流式推理**

AI模型（如GPT-4o）以SSE方式流式返回响应，前端实时展示AI的分析过程和生成的SQL语句。AI生成的SQL示例如下：

    WITH target_dept_ids AS (
        SELECT DISTINCT dept_id
        FROM `auth-cloud`.`sys_user_dept`
        WHERE user_id = (SELECT user_id FROM `auth-cloud`.`sys_user`
                         WHERE nick_name = '张三')
          AND del_flag = 0
    ),
    dept_users AS (
        SELECT DISTINCT user_id
        FROM `auth-cloud`.`sys_user_dept`
        WHERE dept_id IN (SELECT dept_id FROM target_dept_ids)
          AND del_flag = 0
    )
    SELECT
        u.user_id,
        u.nick_name,
        COALESCE(SUM(dd.over_duration), 0) AS total_overtime_hours
    FROM `auth-cloud`.`sys_user` u
    JOIN dept_users du ON u.user_id = du.user_id
    LEFT JOIN `auth-cloud`.`daily_log_detail` dd
        ON u.user_id = (SELECT user_id FROM `auth-cloud`.`sys_user_dept`
                         WHERE id = dd.user_dept_id LIMIT 1)
        AND dd.create_time >= '2026-05-01 00:00:00'
        AND dd.create_time < '2026-06-01 00:00:00'
        AND dd.is_valid = 1
    GROUP BY u.user_id, u.nick_name
    ORDER BY total_overtime_hours DESC;

**步骤6：SQL交互与执行**

前端将SQL代码块渲染为可交互组件，用户点击"Execute"按钮后：
1. 系统将SQL注入编辑器新标签页；
2. 后端对SQL进行安全校验（无危险操作）；
3. 标识符自动修正（含连字符的auth-cloud已正确引用）；
4. 执行SQL并返回结果至虚拟化数据网格展示。
